# 揭秘 SwiftUI 性能（WWDC23）（摘要）

背景：WWDC23 关于构建 SwiftUI 性能心智模型以及分诊卡死/掉帧的讲座。

## 性能循环

- 测量 -> 识别 -> 优化 -> 重新测量。
- 关注具体症状（导航缓慢、动画异常、转圈光标）。

## 依赖和更新

- 视图构成依赖图；动态属性是更新的常见来源。
- 仅在调试时使用 `Self._printChanges()` 检查额外依赖。
- 通过提取视图或收窄状态来消除不必要的依赖。
- 考虑使用 `@Observable` 实现更细粒度的属性追踪。

## 慢速更新的常见原因

- 昂贵的视图体（字符串插值、过滤、格式化）。
- `body` 中的动态属性实例化和状态初始化。
- 列表/表格中缓慢的标识解析。
- 隐性工作：bundle 查找、堆分配、重复字符串构造。

## 避免视图体中的慢初始化

- 不要在视图体中同步创建重度模型。
- 使用 `.task` 获取异步数据并保持 `init` 轻量。

## 列表和表格的标识规则

- 稳定标识对性能和动画至关重要。
- 确保 `ForEach` 中每个元素的视图数量恒定。
- 避免 `ForEach` 中的内联过滤；预过滤并缓存集合。
- 避免在列表行中使用 `AnyView`；它会隐藏标识并增加开销。
- 尽可能扁平化嵌套的 `ForEach` 以减少开销。

## 表格特性

- `TableRow` 解析为单行；行数必须恒定。
- 优先使用精简的 `Table` 初始化器来强制行数恒定。
- 需要时为向后部署使用显式 ID。

## 调试辅助

- 使用 Instruments 分析卡死和掉帧。
- 使用 `_printChanges` 在调试期间验证依赖假设。
