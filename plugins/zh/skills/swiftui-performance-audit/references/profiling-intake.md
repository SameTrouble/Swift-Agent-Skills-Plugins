# 性能分析收集清单

## 用途

当仅靠代码审查无法解释 SwiftUI 性能问题、需要用户提供运行时证据时，使用此清单。

## 首先询问

- 确切症状：CPU 飙升、掉帧、内存增长、卡死，或视图更新过多。
- 确切交互：滚动、输入、初始加载、导航 push/pop、动画、sheet 展示，或后台刷新。
- 目标设备和系统版本。
- 问题是在真机上复现还是仅在模拟器上。
- 构建配置：Debug 还是 Release。
- 用户是否已有基线或变更前后对比。

## 默认性能分析请求

请用户：
- 尽可能以 Release 构建运行应用。
- 使用 SwiftUI Instruments 模板。
- 仅复现确切的问题交互，时长刚好足以捕获问题即可。
- 同时捕获 SwiftUI 时间线和 Time Profiler。
- 导出 trace 或提供关键 SwiftUI 泳道和 Time Profiler 调用树的截图。

## 要求提供这些产物

- 相关 SwiftUI 泳道的 trace 导出或截图
- Time Profiler 调用树截图或导出
- 设备/系统/构建配置
- 简短说明描述捕获时正在进行的操作
- 如果涉及内存问题，提供内存图或 Allocations 数据（如有）

## 何时需要更多

- 如果首次运行混合了多个交互，要求第二次捕获。
- 如果用户已尝试修复，要求变更前后对比。
- 如果问题仅在模拟器出现或滚动流畅度很重要，要求真机捕获。

## 常见陷阱

- Debug 构建会扭曲 SwiftUI 的计时和分配行为。
- 模拟器 trace 可能遗漏设备独有的渲染或内存问题。
- 一次捕获中混合多个交互会使归因更困难。
- 没有复现说明的截图更难解读。
