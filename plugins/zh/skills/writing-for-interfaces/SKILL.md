# 为界面而写

好的界面文案是隐形的。当文字与设计无缝协作时，人们注意不到它们的存在。

写作应从一开始就是设计流程的一部分，而不是最后填补的空白。当文字与布局、交互和视觉设计同步考量时，成果会感觉浑然一体。当它们被当作事后补充时，产品体验就会显得拼凑而生硬。

界面中的每一段文字都是一次微小的沟通行为：它应尊重对方的时间，在对方所处的位置与之相遇，并帮助他们继续前行。

## 触发条件

### 第一步：确立语调与个性

语调（voice）是基础。所有文案决策——说什么、怎么说、不说什么——都源自对这个问题的清晰理解：这个产品是谁、为谁服务、应该听起来怎样。没有定义好的语调，文案就会前后不一致，产品也会失去整体感。

**搜索现有的语调定义。** 检查以下来源：

- `CLAUDE.md`、`AGENTS.md` 或类似的项目配置，是否定义了语调和/或语气
- 风格指南、设计系统文档或品牌规范
- 术语表或用词参考

**如果已存在语调定义**，将其作为所有文案工作的滤镜。如果你正在处理的文案与它偏离，标记出不一致之处。

**如果不存在语调定义**，从现有文案中推断当前语调。寻找规律：正式还是随意？技术化还是平实？温暖还是就事论事？如果文案不一致或样本不足以推断，帮助用户在动笔前先确立语调。

#### 通过对话确立语调

引导用户回答这些问题：

1. **这个产品做什么，为谁而做？** 面向职场人士的银行 App 和面向孩子的储蓄 App 服务相似目的，但听起来应完全不同。受众决定了词汇、复杂度和语体。

2. **人们为什么用它，在哪里用？** 在危机中使用健康 App 的人需要平静清晰的表述。在家中浏览游戏的人可以接受活泼的语气。使用情境——物理环境、情绪状态、注意力竞争——决定了人们能吸收多少文字、什么语气合适。

3. **把产品想象成一个人。哪 3–4 个性格特质让它独一无二？** 自由头脑风暴，把相似的词归为主题，丢弃"理所当然"的特质（"不让人困惑"），保留那些真正区分产品个性的特质。

4. **寻找有生产力的张力。** 最好的语调定义中，存在互相推挤的特质。"友善"和"简洁"形成有用的张力——这些就成了你在不同情境下调整语气时拨动的旋钮。

5. **记录下来。** 建议用户把语调定义持久化到可靠的地方（`AGENTS.md`、`CLAUDE.md` 或风格指南文档），使其跨会话保留。术语表与之很搭，应存放在同一文件中。

### 第二步：评估请求

确定需要哪种文案工作：

- **新文案**：从零开始为屏幕、流程或组件撰写。
- **审查**：评估现有文案的清晰度、一致性和语气。
- **重写**：改进那些不奏效的具体文字。
- **术语**：构建或维护术语表。

然后识别涉及哪些界面模式，并查阅 `references/patterns.md` 中相关章节。

### 第三步：先应用语调，再应用原则

对每一段文案，按此顺序工作：

1. **听起来像这个语调吗？** 对照那 3–4 个特质朗读。如果你大声读出来，能认出它出自这个产品吗？
2. **这个情境下，哪些特质需要调高或调低？** 把每个语调特质想象成一个旋钮。庆祝时刻调高温暖；出错时调高清晰度、调低友善。
3. **应用核心原则**（目的、预判、情境、同理心——详见下文）。
4. **应用写作技法规则**（删除填充词、避免重复、具体明确——详见下文）。

这个顺序是刻意设计的，编码了一条优先级链：**清晰 > 语调 > 技法规则。** 清晰永远胜出——如果语调妨碍了某人理解该做什么，就剥除它。语调其次——它塑造声音，技法规则永远不应以削弱既定语调的方式删词或改写句式。技法规则是经过语调过滤的启发式经验，不是绝对法则。提交前始终要把技法编辑与语调交叉核对。

### 第四步：交付修改

逐个元素处理文案——标题、正文、按钮、标签——先展示原文，再展示改写，附上简短理由，关联到语调和原则。优先处理让用户困惑或受阻的修改，再做润色。在跨流程审查时，标记术语不一致，并在末尾建议术语表条目。

## 语调与语气

### 语调 vs. 语气

**语调（voice）** 是产品一贯的个性——定义它始终如何发声的 3–4 个特质。这些不变。

**语气（tone）** 是语调如何随情境调整。把每个语调特质想象成可随时刻上下拨动的旋钮：

- 庆祝里程碑？调高温暖，调低简洁。
- 报告错误？调高清晰和有用度，调低友善。
- 新用户引导？在有用度与温暖之间取平衡。
- 确认破坏性操作？调高直接，保持冷静简洁。

### 在实践中应用语气

针对每个情境，决定哪些语调特质需要强调、哪些应该退后。

**示例**：当某人无法连接网络时，清晰和有用度大幅上调。简洁保持中等，因为他们需要最重要的细节。友善下调，因为帮他们脱困比听起来温暖更重要。

### 个性该放在哪里

个性在有余地的时刻闪耀——欢迎屏、里程碑、空状态。在错误消息、破坏性操作和关键流程中，收起语调，让清晰主导。第三步的优先级链适用：清晰永远优先。

## 核心原则

目的、预判、情境、同理心——一个关于写什么、怎么写、何时写的框架。透过你的语调滤镜应用。

### 1. 目的

动笔前先回答：**此刻这个人最需要知道的那一件事是什么？**

- **使用信息层级。** 标题和按钮承载主要信息；辅助文字补充细节。如果某人只读标题和按钮，就应理解当前情况。
- **删去不为此刻服务的内容。** 挪到别处或移除。当一个屏幕试图承载太多，回到它的目的，剥除其余一切。
- **告诉人们目的。** 引入功能时，告诉他们它为何存在、为何对他们重要。

### 2. 预判

把界面想象成一场对话。任何好的对话都有自然的往复——倾听、回应、预判对方接下来需要听到什么。

- 告诉某人一个问题之后，告诉他怎么修。"无法连接 Wi-Fi" → "无法连接 Wi-Fi。请检查你的连接后重试。"
- 要求某人做某事之后，让做法显而易见。"验证你的身份" 配上清晰的按钮或链接来启动流程。
- 某人完成某事后，确认它并指明下一步。"密码已更改" → "密码已更改。你现在可以用新密码登录了。"
- **先讲"为什么"。** 把好处或理由放在指令之前："为了 [好处]，[指令]。" 先给动机让指令像是合理的请求，而非命令。

### 3. 情境

人们在截然不同的境况下使用产品。使用情境塑造写作。

- **跳出应用本身思考。** 考量物理与情绪状态。
- **密度匹配可用注意力。** 任务中文字应极简；安装流程可以容纳更多。
- **时机重要。** 在相关时刻才展示信息，而非提前。把说明放在人们视线所在之处。
- **为设备而写。** 正确描述手势（触屏用"轻点"而非"点击"）。手机要求简洁；共享屏幕（电视）需要大而可扫读的文字。

### 4. 同理心

为所有可能使用这个产品的人而写——不同的能力、语言、文化、技术熟练度和情绪状态。

- **使用平实、直接的语言。** 避免行话、成语和文化特定的引用。
- **从一开始就为无障碍设计。** 标签、描述和替代文本不是事后补充——对某些人来说它们就是全部体验。详见模式参考。
- **使用包容、中性的语言。** 避免不必要的性别、年龄或能力指涉。
- **考虑本地化。** 写短文案，而非压缩的长文案。预留文本扩展和 RTL 语言的空间。

## 写作技法

收紧文案的实用编辑动作。在确认语调和语气正确之后应用。

### 删除填充词

界面文字没有最低字数要求。每个词都必须有理由存在。但删词前，先检查它是否在做语调工作。按通用技法规则算"填充"的词，对语调可能是承重的——"暂无内容"里的"暂"承载温暖与从容，删掉它会让空状态更生硬。**测试方法**：删掉这个词；若意义和刻意追求的语气都没变，就删。

- **副词/形容词**："只需输入车牌号" → "输入车牌号。""只需""快速""轻松""就""成功"这类词常在承诺你无法保证的东西。保留那些真正澄清的词（"自动喂养你的宠物"）。
- **感叹语**：错误消息里的"哎呀！""糟糕！"会淡化问题。删掉。
- **客套语**：自动消息里的"抱歉"和"请"听起来虚伪。仅当真正增加温暖时才用。
- **标点**：
  - **感叹号**：罕见。留给真正值得庆祝的时刻。
  - **破折号**（en/em）：界面文案中避免使用。它们打断扫读。改成分行或分句。
  - **省略号**：仅用于进行中的流程（"加载中…"），不要用于悬而未决的思绪。

### 避免重复

把重叠的想法合并成一条清晰的表述。屏幕上的每个元素都应带来新信息。当标题和正文用不同的话说同一件事，就合并它们。

"我们要迟到了。你的外卖骑手赶不上了。他们 10 分钟内到。" → "配送延迟 10 分钟。在 App 中查看骑手位置。"

### 具体而非含糊

- 点名事物：用"无法打开'季度报告.pdf'"，而非"无法打开此文件"。
- 点名动作：用"取消订阅"/"保留订阅"，而非"是"/"否"。
- 给出真实信息：用"尾号 4242 的卡被拒"，而非"出现支付错误"。

### 维护术语表

决定你管东西叫什么，并坚持。一个屏幕叫"别名"了，别在另一个屏幕叫"用户名"。术语表是一张简单的表：**用** / **不用** / **定义**。按钮标签尤其适合入表——如果"下一步"用来推进流程，那到处都用"下一步"。

### 代词与视角

"收藏"传达的信息和"你的收藏"一样。避免"我们"——它掩盖了实际发生了什么（"我们遇到问题…" → "无法加载内容"）。

### 死磕细节

正确的拼写、语法和标点建立信任。采用与语调匹配的大小写规则（标题式 = 正式，句式 = 随意）并一致应用。为可用空间而写——如果文案需要短，就写一个短句，别压缩一个长句。

### 为动态内容而写

模板化字符串（`"${count} 项已选"`）也是界面文案。要让每种可能的输出都读起来自然：

- **处理零、一和多数。** "无结果""1 项结果""24 项结果"——而不是一个产出"找到 0 个结果"的模板。
- **用真实值读一遍。** 代入短名和长名、小数和大数。"欢迎回来，克里斯托弗-蒙哥马利！"可能撑爆布局；"3 秒前"和"2 天前"都应读起来自然。
- **保持模板简单。** 如果一个字符串需要复杂分支才读得顺，可能是设计对单个元素要求过高。

### 构建语言模式

为常见时刻定义模式——流程如何开始（"开始使用"）、推进（"下一步"或"继续"——选一个）、结束（"完成"）。一致地使用它们。

## 最简单的测试

大声朗读你的文案。如果它听起来像你在向朋友解释一件事——清晰、自然、没有填充——那大概就不错。如果它听起来像机器人、法律文件或作文，继续打磨。
