# Swift API 设计指南技能

## 概述
使用本技能来设计和审查 Swift API，使其在使用点清晰、在调用处流畅，并与既有的 Swift 命名和标签约定保持一致。优先考虑可读性、显式意图，以及在声明、调用处和文档注释之间的一致性。

## 工作决策树

### 1) 审查既有代码
- 同时检查声明和调用处，而非只看声明。
- 检查命名的清晰度与流畅性（见 `references/promote-clear-usage.md`、`references/strive-for-fluent-usage.md`）。
- 检查参数标签和参数命名（见 `references/parameters.md`、`references/argument-labels.md`）。
- 检查文档注释和符号标记（见 `references/fundamentals.md`）。
- 检查约定和重载安全性（见 `references/general-conventions.md`、`references/special-instructions.md`）。

### 2) 改进既有代码
- 重命名含糊、冗余或角色不清的 API。
- 重构标签以改善调用处的语法阅读。
- 用基于角色的名称替换弱命名参数。
- 解决因弱类型而变得含糊的重载集合。
- 强化文档摘要，以精确描述行为和返回值。

### 3) 实现新功能
- 在最终确定声明之前，先从使用点示例入手。
- 选择基名和标签，使调用读起来像清晰的英文短语。
- 仅在能简化常见用法时添加默认值。
- 以一致的命名定义 mutating/nonmutating 配对。
- 为每个新声明添加简洁的文档注释。

## 核心指南

### 基本原则
- 使用点的清晰度是最高优先级。
- 清晰度比简洁性更重要。
- 每个声明都应有文档注释。
- 摘要应说明声明做什么、返回什么、访问什么、创建什么或是什么。
- 使用公认的 Swift 符号标记（`Parameter`、`Returns`、`Throws`、`Note` 等）。

### 促进清晰使用
- 包含所有避免歧义所需的词。
- 省略不必要的词，尤其是类型重复。
- 按角色而非类型命名参数和关联类型。
- 当类型信息较弱（`Any`、`NSObject`、`String`、`Int`）时，添加角色名词。

### 力求流畅使用
- 优先选择能产生语法正确、可读调用处的方法名。
- 工厂方法以 `make` 开头。
- 无副作用的 API 命名为名词短语；有副作用的 API 命名为祈使动词。
- 保持 mutating/nonmutating 命名配对一致（`sort`/`sorted`、`formUnion`/`union`）。
- 布尔 API 应读起来像断言（`isEmpty`、`intersects`）。

### 善用术语
- 除非为精确性需要专业术语，否则优先使用常见词汇。
- 如果使用专业术语，保留其既定含义。
- 避免非标准缩写。
- 当既有领域惯例能增进共同理解时，予以采纳。

### 约定、参数与标签
- 为非 `O(1)` 的计算属性文档化其复杂度。
- 除特殊情况外，优先使用方法/属性而非自由函数。
- 遵循 Swift 大小写约定，包括首字母缩略词处理。
- 使用能提升生成文档可读性的参数名。
- 当语义共享时，优先使用默认参数而非方法族。
- 将带默认值的参数放在末尾附近。
- 根据语法和含义应用参数标签，而非风格偏好。

### 特殊指令
- 在公共 API 表面中为元组成员添加标签并为闭包参数命名。
- 对无约束多态要显式处理，以避免重载歧义。
- 使名称与文档注释中展示的语义保持一致。

## 快速参考

### 命名形态
| 情形 | 首选模式 |
| --- | --- |
| Mutating 动词 | `reverse()` |
| Nonmutating 动词 | `reversed()` / `strippingNewlines()` |
| Nonmutating 名词操作 | `union(_:)` |
| Mutating 名词操作 | `formUnion(_:)` |
| 工厂方法 | `makeWidget(...)` |
| 布尔查询 | `isEmpty`、`intersects(_:)` |

### 参数标签规则
| 情形 | 规则 |
| --- | --- |
| 可区分的无标签参数 | 仅在区分仍然清晰时省略标签 |
| 保值转换初始化器 | 省略第一个标签 |
| 介词短语中的第一个参数 | 通常从介词取标签 |
| 语法短语中的第一个参数 | 省略第一个标签 |
| 带默认值的参数 | 保留标签（在调用处可省略） |
| 所有其他参数 | 加上标签 |

### 文档规则
| 声明种类 | 摘要应描述 |
| --- | --- |
| 函数 / 方法 | 做什么以及返回什么 |
| 下标 | 访问什么 |
| 初始化器 | 创建什么 |
| 其他声明 | 是什么 |

## 审查清单

### 清晰度与流畅性
- [ ] 调用处无需阅读实现细节即可清晰理解。
- [ ] 基名包含消除歧义所需的所有词。
- [ ] 命名简洁，避免重复类型名。
- [ ] 在最重要的地方，调用读起来自然且符合语法。

### 命名语义
- [ ] 无副作用的 API 读起来像名词/查询。
- [ ] 有副作用的 API 读起来像祈使动词。
- [ ] mutating/nonmutating 配对使用一致的命名模式。
- [ ] 布尔 API 读起来像断言。

### 参数与标签
- [ ] 参数名提升文档和角色清晰度。
- [ ] 默认参数简化常见用法。
- [ ] 带默认值的参数位于末尾附近。
- [ ] 第一个参数标签遵循语法和转换规则。
- [ ] 其余参数加标签，除非有明确理由省略。

### 文档与约定
- [ ] 每个声明都有有用的摘要注释。
- [ ] 在合适处使用符号标记。
- [ ] 非 `O(1)` 的计算属性复杂度已文档化。
- [ ] 大小写约定和首字母缩略词大小写遵循 Swift 规范。
- [ ] 重载避免仅靠返回类型区分和弱类型歧义。

## 哲学
- 优先考虑清晰的使用点语义，而非声明上的巧妙。
- 在发明本地风格规则之前，先遵循既有的 Swift 约定。
- 为公共 API 表面的可维护性和可审查性进行优化。
- 保持指导实用：应用能提升清晰度的最小改动。
