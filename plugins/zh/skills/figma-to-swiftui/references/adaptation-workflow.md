# 适配工作流

将现有 SwiftUI 屏幕适配以匹配更新后 Figma 设计的指南。当用户要求更新、适配或对齐现有屏幕到 Figma 设计时，此工作流替代标准的"从零构建"方法。

## 何时使用

在以下情况触发此工作流：
- 用户说"适配"、"更新"、"对齐"、"匹配"现有屏幕到 Figma 设计
- 用户提供 Figma URL 并引用代码库中的现有视图/屏幕
- 任务是修改现有代码而非创建新视图

## 适配审计流程

### 1. 阅读现有代码

阅读正在适配的视图的完整源代码，包括：
- 主视图文件
- 它引用的任何子组件（自定义视图、共享组件）
- 相关模型类型以理解可用数据

注意每个元素及其属性：间距、padding、颜色、字体、布局结构、圆角、不透明度值。

### 2. 构建差异清单

将现有代码与 Figma 设计上下文和截图**逐元素**比较。将每个差异分类：

- **ADD** —— 元素在 Figma 中存在但代码中没有
- **UPDATE** —— 元素在两者中都存在但属性不同。始终包含旧 → 新值。
- **REMOVE** —— 元素在代码中存在但 Figma 中没有。删除前始终与用户确认。

### 3. 间距审计

间距是最常被遗漏的差异。对于每个容器和元素，明确比较：
- 水平和垂直 padding 值
- 栈间距值（VStack/HStack spacing 参数）
- 元素间间距
- 边缘缩进和安全区域处理
- Frame 尺寸（宽度、高度）

绝不假设现有值"足够接近"。如果 Figma 说 20 而代码说 16，那是必须列出的更改。

### 4. 展示清单

在编写任何代码之前向用户展示完整清单。使用此格式：

```
发现的差异：

### 结构
- ADD：计时器卡片组件（青柠背景、倒计时、进度条）
- ADD：带文本叠加的插画头部
- REMOVE：独立的获胜者区块——确认？

### 布局与间距
- UPDATE：头像尺寸 56 → 64
- UPDATE：卡片间距（头像 ↔ 内容）12 → 8
- UPDATE：每张卡片底部 padding 16 → 24
- UPDATE：分隔符不透明度 0.12 → 0.14

### 排版
- UPDATE：标题字体 17pt medium → 20pt semibold
- UPDATE：队伍名字体 17pt → 20pt regular
- UPDATE：积分字体 28pt semibold → 22pt expanded semibold

### 颜色与样式
- UPDATE：名次徽章——金/银/铜渐变 → 全部紫色渐变
- UPDATE：背景渐变——硬编码 RGB → asset catalog 颜色

### 新数据需求
- 计时器：硬编码还是需要 API 数据？
- 统计标签：需要数据源？
```

按类别分组更改（结构、间距、排版、颜色），使用户可以系统地审查。

### 5. 澄清未知项

在实现之前，询问用户关于：
- **新组件**需要当前模型中不可用的数据（例如计时器、统计）
- **已移除元素**——删除前确认
- **模糊元素**——当 Figma 显示可能是系统提供或自定义的内容时

### 6. 应用所有更改

用户确认后，应用清单中的每一项。不要跳过看似次要的项——4px 的 padding 差异或 0.02 的不透明度变化对视觉保真度很重要。

## 常见陷阱

1. **"足够接近"偏见** —— 当现有代码看起来与设计相似时，很容易跳过小差异。清单防止了这一点。
2. **遗漏新元素** —— 专注于现有元素的更改可能导致你忽略设计中添加的全新组件。
3. **忽略已移除元素** —— 如果设计不再显示代码有的内容，标记它以移除，而非留下死 UI。
4. **间距捷径** —— 绝不目测间距。从 `get_design_context` 属性（padding、gap、itemSpacing）提取精确值。
5. **字重/字宽混淆** —— Figma "Expanded Semibold" 是 `.semibold` 字重 + `.width(.expanded)`，不仅仅是 `.semibold`。检查字重和字宽两者。
