# 屏幕发现

当提供的 Figma 节点不明显是单个屏幕或组件时使用此工作流。

## 何时触发

当输入是以下情况时，在 `get_design_context` 之前运行屏幕发现：
- 根节点，如 `0:1`
- 页面节点
- 包含多个子画板的大型 frame
- 似乎容纳多个屏幕、引导步骤、设备变体或状态变体的容器
- 源文档提到的屏幕数多于提供的 Figma URL 明显包含的屏幕数

## 目标

在获取昂贵的设计上下文或编写代码之前，将模糊的 Figma 容器转换为简短的、可审查的屏幕映射。

## 必需步骤

1. 在提供的节点上运行 `get_metadata`。
2. 识别可能的屏幕画板或顶层区块。
3. 构建带置信度的候选映射表。
4. 仅当屏幕边界足够明确时继续。

## 必需输出

在获取或编码之前，产生类似这样的紧凑表格：

```text
候选屏幕映射：
- Splash -> 节点 12:4 -> 置信度：高
- Intro 1 -> 节点 12:8 -> 置信度：中
- Intro 2 -> 节点 12:12 -> 置信度：中
- Main -> 节点 12:20 -> 置信度：高

模糊点：
- 节点 12:8 和 12:9 可能都是 Intro 1 变体

建议：
- 使用 12:8 作为默认实现节点
```

## 置信度规则

- `高`：画板名称、结构或可见内容明确映射到一个屏幕
- `中`：画板是可能的匹配，但有竞争候选或标签不明确
- `低`：多个候选都合理，或节点可能根本不是屏幕

## 停止条件

在以下情况时停止并在实现之前询问用户：
- 任何关键屏幕仅以`低`置信度映射
- 两个或更多候选会实质性改变生成的 UI 或流程
- 节点似乎在同一层级混合了屏幕状态和屏幕
- 源文档和 Figma 结构在屏幕数量上不一致

## 默认规则

当根节点的另一个子节点是同一屏幕的可能匹配时，不要默默选择根节点的某个子画板。
