# 获取策略

以避免超时、最小化浪费令牌、并保持工作可恢复的方式调用 Figma MCP 工具的规则。

## 核心原则

1. **元数据先于上下文**：`get_design_context` 是昂贵、易超时的调用。除非确信节点是单个可实现单元，否则不要对节点调用它。
2. **先用源文档缩小范围**：当需求文档或工单存在时，使用其屏幕列表和范围外说明来决定哪些 Figma 节点重要。
3. **按文件和节点去重**：`get_variable_defs` 按 `fileKey`；资源按源节点 ID。
4. **拆分而非盲目重试**：如果上下文太大或被截断，使用元数据获取更小的有意义区块。

## 元数据优先决策树

```text
输入节点
├── 明显是一个屏幕/组件
│   └── 直接获取设计上下文
├── 根 / 页面 / 可能是多屏幕容器
│   └── 先运行 get_metadata
│       ├── 一个明确的子屏幕 -> 重定向到该子节点
│       └── 多个候选 -> 构建屏幕映射，如果模糊则询问
└── 未知
    └── 先运行 get_metadata
```

## 何时强制屏幕发现

在以下情况时，在 `get_design_context` 之前运行 `get_metadata`：
- 节点 ID 是文件级/根级，如 `0:1`
- 节点名称包含 `Page`、`Flow`、`Onboarding`、`All Screens`、`Desktop`、`iPhone & iPad`
- 节点有许多直接子节点或深层嵌套
- 用户提供了根/页面 URL
- 源文档提到的屏幕数多于 URL 明显包含的屏幕数

候选映射格式参见 references/screen-discovery.md。

## 大型或截断上下文的熔断器

如果 `get_design_context` 超时或返回截断输出：

1. 不要用相同参数重试同一节点。
2. 在该节点上运行 `get_metadata`。
3. 选择与目标区块匹配的最小有意义子节点。
4. 在该子节点上获取 `get_design_context`。
5. 如果仍然太大，拆分为区块，如头部、主体、底部、卡片或模态内容。
6. 当你拆分屏幕时告诉用户，让他们理解为什么使用多个节点上下文。

## 去重令牌获取

`get_variable_defs(fileKey, nodeId)` 返回文件范围的变量。如果多个屏幕来自同一 `fileKey`，获取一次变量并为其余屏幕复用结果。

当所有屏幕共享同一 Figma 文件时，不要每屏调用一次 `get_variable_defs`。

## 按源节点去重资源

当同一图标或插画源节点出现在多个屏幕上时，获取一次并复用。

推荐的缓存结构：

```text
.figma-cache/
  _shared/assets/
    3166_70211.png
    3166_70211.meta.json
```

使用源节点 ID 作为稳定键，将 `:` 替换为 `_` 作为文件名。

## 调用预算健全性检查

对于单个屏幕，正常模式是：
- `get_metadata`：上下文之前 0-1 次，如果上下文超时或资源需要节点 ID 则加 1 次
- `get_design_context`：每屏 1 次，仅在拆分后有区块调用
- `get_screenshot`：屏幕 1 次加上每个扁平化区域或图标资源 1 次
- `get_variable_defs`：每个 `fileKey` 1 次
- Code Connect 查找：可选，仅当 MCP 服务器暴露它时

超出此范围是停止并重新规划的信号，而非继续盲目获取。

## 不应做的事

- 不要为了看看里面有什么而对根/页面/流程容器调用 `get_design_context`。
- 不要用同一节点重试超时的上下文调用。
- 不要为同一 Figma 文件中的每个屏幕重新获取变量。
- 当同一图标重复出现时，不要跳过资源节点去重。
