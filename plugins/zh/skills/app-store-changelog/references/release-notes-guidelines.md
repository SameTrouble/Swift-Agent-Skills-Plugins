# App Store 发布说明指南

## 目标
- 生成面向用户的发布说明，描述自上一个 tag 以来的可见变化。
- 包含所有影响用户的变更；去除纯内部或仅为重构的工作。
- 语言通俗、简短，聚焦收益。

## 输出形态
- 大多数发布控制在 5 到 10 条条目。
- 如需可按主题分组：新增、改进、修复。
- 每条一句话，以动词开头。
- 避免内部代号、工单 ID 或文件路径。

## 筛选规则
- 包含：新功能、UI 变更、行为变更、用户会注意到的 bug 修复、有可见影响的性能改进。
- 排除：重构、依赖升级、CI 变更、开发工具、内部日志、分析变更，除非其影响用户隐私或行为。
- 如果某条变更含义模糊，请请求澄清，或仅在其对用户可见时描述为一项小改进。

## 语言指引
- 将技术术语转化为面向用户的描述。
- 避免出现"API"、"refactor"、"nil"、"crash log"、"dependency"等词。
- 优先使用"改进"、"新增"、"修复"、"更新"，或"搜索"、"上传"、"同步"等动作动词。
- 保持时态统一（现在时或过去时）：如"新增"、"改进"、"修复"。

## 示例
- "新增了从个人资料菜单切换账号的功能。"
- "改进了弱网环境下时间线的加载速度。"
- "修复了媒体附件无法全屏打开的问题。"

## QA 清单
- 每一条都对应该范围内的一项真实变更。
- 不存在描述同一变更的重复条目。
- 不含内部行话或文件路径。
- 最终列表符合所提供目标上架地区的 App Store 文本字数限制。
