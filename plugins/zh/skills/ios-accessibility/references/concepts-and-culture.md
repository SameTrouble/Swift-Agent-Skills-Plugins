## 无障碍是非确定性的

没有 universally "正确"的无障碍体验。无障碍关乎用户及其上下文：
- 不同的残障需要不同的适配
- 同一用户在不同情况下可能有不同的需求
- 不同的方法对不同的辅助技术可能有不同的权衡

**良好实践是默认值，而非绝对值。** 用真实用户和辅助技术验证，然后根据反馈改进。

### 从同理心开始

如果一个流程用 VoiceOver 或 Switch Control 很难使用，通常视觉上也会很难。无障碍测试可以更早地发现一般性 UX 问题。

### 四个无障碍领域

Apple 将其无障碍功能分为四个类别：
- **视觉**（失明、低视力、色盲）
- **听觉**（失聪、听力受损）
- **运动**（灵活度受限、震颤）
- **认知**（学习、注意力、记忆）

## 包容性设计

包容性设计从一开始就考虑全范围的人类多样性——而不是事后补充。

### 原则

1. **识别排斥**：确定谁被排除在外以及为什么
2. **为一人解决，扩展到多人**：针对特定残障的解决方案通常惠及所有人
3. **从多样性中学习**：尽可能让残障人士参与

### 语言和命名

名称和标签会影响产品的包容感。

- 优先使用中性、描述性的名称，而非带有能力负担的标签
- 避免根据访问需求将用户分为"高级"与"受限"
- 保持选项名称与行为具体相关（例如，"引导模式"、"宽松计时"、"高对比度提示"）

### 示例

路缘坡道是为轮椅用户设计的，但也惠及推婴儿车的父母、带行李的旅客以及推车的配送工人。

## 残障模型

### 医学模型

将残障视为个体需要修复的缺陷。

### 社会模型

将残障视为人与环境之间的不匹配。障碍是外部的，而非内部的。

无障碍工作通常与社会模型一致：移除产品中的障碍，而非"修复"用户。

## 无障碍是一种实践

> "无障碍既是一种状态，也是一种实践。" — Sommer Panage

将无障碍视为一次性审计会导致回归。将无障碍嵌入设计和工程系统才能创造持久的改进。

### 结果优先于一次性产出

无障碍待办列表或审计报告很有用，但不是目标。目标是用户使用辅助技术成功完成真实任务。

将审计作为优先级排序和规划的输入，而非终点。

### 持续的习惯

- 在设计阶段就考虑无障碍，而不仅仅是最后（左移）
- 改进共享组件以扩大影响
- 自动化用于护栏，而非完全覆盖
- 定期用辅助技术测试

### 按影响排优先级

当时间有限时：

- 从顶级用户旅程开始（引导、认证、结账、媒体播放、设置）
- 先修复阻塞问题，再处理润色
- 优先处理共享组件和设计令牌以获得广泛影响
- 迭代发布，然后重新测试和改进

## 文化与盟友

无障碍专家无法独自完成。建立文化意味着：
- 培训设计师和工程师掌握基础知识
- 创建安全空间（Slack 频道、办公时间）供提问
- 庆祝胜利并认可贡献
- 将无障碍嵌入入职培训

### 领导力和认同

文化变革在自下而上和自上而下的支持下效果最佳：

- 自下而上：倡导者展示实际修复并指导他人
- 自上而下：领导层保护时间并将无障碍纳入质量期望
- 沟通：用用户结果和产品风险来解释影响，而不仅仅是合规语言

## 常见误解

| 误解 | 现实 |
|---------------|---------|
| "无障碍是关于改善 VoiceOver 用户的体验" | 有许多辅助技术、无障碍功能和良好实践需要考虑 |
| "无障碍很昂贵" | 改造很昂贵；从一开始就构建无障碍并不昂贵 |
| "自动化能覆盖一切" | 并非所有问题都可检测；无障碍通常需要判断 |
| "合规即可用" | 通过审计并不能保证良好的体验 |

## 左移

在流程中更早地处理无障碍：

| 阶段 | 行动 |
|-------|--------|
| 设计 | 在原型上标注标签、特质...考虑颜色对比度，界面在无障碍文本大小下如何工作 |
| 开发 | 边构建边实现无障碍 |
| 代码审查 | 检查无障碍属性 |
| 测试 | 用辅助技术手动测试 |
| 审计 | 验证，而不仅仅是发现 |

问题发现得越晚，修复成本越高。

## 跨辅助技术的收益

像 `accessibilityLabel`、`accessibilityTraits` 和自定义操作这样的 API 惠及多种技术：
- VoiceOver
- Voice Control
- Switch Control
- Full Keyboard Access
- 盲文显示器

一次实现，广泛受益。从 VoiceOver 开始——它涵盖了大部分基础知识。

## 组织实践

### 倡导者网络

培训跨团队的倡导者，他们可以回答基本问题并升级复杂问题。对于较大的组织，轻量级的无障碍协会可以协调跨团队工作。

### 无障碍内容

无障碍不仅仅是代码。文档、营销和支持内容也需要关注。

### 发布说明

在发布说明中突出无障碍改进。这表明了承诺并邀请依赖这些功能的用户提供反馈。

### 入职和培训

在入职材料中包含无障碍，让新团队成员了解它是质量定义的一部分。

### 完成定义

在发布功能的清单中添加**"无障碍"**。可以简单到："已用 VoiceOver 和 Dynamic Type 测试。"

### 研讨会和午餐学习会

在团队中分享改进和经验教训。真实修复的简短演示通常最有效。作为团队观看 *"Convenience for You is Independence for Me"*（WWDC 2017）——Todd Stabelfeldt 关于四肢瘫痪生活以及应用如何改变他独立性的故事一直很有影响力。

### 代码审查和 QA

鼓励审查者尽早标记无障碍问题。检出分支并运行应用——在审查时发现 bug 比发布后快得多。在 QA 冒烟检查中添加快速手动测试（VoiceOver、Dynamic Type）。

在产品讨论和设计审查中，持续追问：*"在最大的无障碍字体大小下这看起来怎样？是否有任何关键操作隐藏在手势后面？我们是否为所有交互元素提供了文案？"* 经过几次重复，团队就会开始自己问这些问题。

### 招聘和职场

残障人士在招聘流程和职场工具中面临障碍。改进这些方面能创建更包容的团队，从而构建更包容的产品。考虑在 iOS 开发者职位描述中将无障碍知识作为要求或积极加分项——这能在社区规模上提高认知。

### 审计文档

一个实用的起步方法是戴上耳机，打开 VoiceOver，在应用中浏览最重要的流程。在共享文档中记录你的发现——按屏幕或功能组织。保留已完成的修复，而不是删除它们（划掉或添加 ✅）。这有两个目的：它直观地传达当前状态（长列表是明确的信号），并在队友询问"我能帮什么忙？"时给他们一个直接的起点。

一旦你有了列表，在冲刺演示、全员大会或内部聚会中分享。展示一个特别糟糕的体验，然后展示修复后的样子。大多数修复只需要几行代码。直接看到这一点，任何人都很难争辩说这太复杂或太昂贵。

用辅助技术演示，以提高团队其他成员对用户与产品交互方式多样性的认知。

### 可持续性和防止倦怠

无障碍工作是长期的。避免英雄文化并分散所有权：

- 定义现实的迭代目标，而不是"现在就修复一切"
- 庆祝增量胜利
- 保留决策日志，以便在人员换团队时知识得以延续
