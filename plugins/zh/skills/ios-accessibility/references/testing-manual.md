# 手动测试

使用辅助技术和设备设置测试 iOS 无障碍。

## 为什么要手动测试

自动化工具能捕获基本问题，但无法评估：
- 标签在上下文中是否有意义
- 导航顺序是否遵循合理流程
- 体验是否真正可用
- 复杂交互在辅助技术下的感受

始终将自动化检查与实际操作测试结合。

## 快速设置

### 无障碍快捷方式

三次点击侧边（或主屏幕）按钮以切换辅助技术。

在**设置 > 无障碍 > 无障碍快捷方式**中配置。

推荐：添加 VoiceOver、Voice Control 和 Full Keyboard Access。

提示：选择少量快捷方式，这样你可以快速切换而无需循环太多选项。

### 控制中心

将**无障碍快捷方式**添加到控制中心以更快切换。

**设置 > 控制中心 > 自定义控制 > 无障碍快捷方式**

### Siri

- "Turn on VoiceOver"
- "Turn off VoiceOver"
- "Turn on Voice Control"

## VoiceOver 测试

### 核心手势

| 手势 | 操作 |
|---------|--------|
| 轻点 | 选择元素 |
| 双击 | 激活 |
| 向右滑动 | 下一个元素 |
| 向左滑动 | 上一个元素 |
| 双指旋转 | 更改转子模式 |
| 上下滑动 | 调节（如可调节）或按转子导航 |
| 双指双击 | Magic Tap |
| 双指擦洗（Z 形） | Escape |
| 三指滑动 | 滚动 |

### 屏幕变暗

三指三次点击以在 VoiceOver 保持活动时将屏幕变黑。强制非视觉测试。

使用屏幕变暗验证核心流程在没有视觉提示的情况下工作。如果在屏幕变暗下感觉困难，UI 可能需要简化或更好的结构。

### 字幕面板

在**设置 > 无障碍 > VoiceOver > 字幕面板**中启用，以在屏幕底部查看 VoiceOver 输出。

### 验证内容

1. 每个交互元素都可通过滑动到达
2. 标签清晰描述元素
3. 特质与角色匹配（按钮、标题等）
4. 导航顺序遵循任务流程
5. 状态变化后焦点移动到新内容
6. 错误被公告
7. 自定义手势有无障碍替代方案

### 高级手势

| 手势 | 操作 |
|---------|--------|
| 双指单击 | 暂停/恢复语音 |
| 三指双击 | 静音/取消静音 VoiceOver 语音 |
| 单指三次点击 | 长按 |
| 四指点击顶部 | 第一个元素 |
| 四指点击底部 | 最后一个元素 |

## Voice Control 测试

### 启用

**设置 > 无障碍 > Voice Control**

或对 Siri 说"Turn on Voice Control"。

### 测试

1. 说"Show names"以覆盖无障碍标签
2. 尝试说出按钮标签以激活它们
3. 说"Show grid"以精确点击
4. 对元素说"Show actions"以查看自定义操作

### 验证内容

1. 标签与可见文本匹配（或直观）
2. 不同控件没有重复标签
3. 所有交互元素都可通过语音激活
4. 自定义操作可发现

## Full Keyboard Access 测试

### 启用

**设置 > 无障碍 > 键盘 > Full Keyboard Access**

### 导航

| 按键 | 操作 |
|-----|--------|
| Tab | 下一个元素 |
| Shift + Tab | 上一个元素 |
| Space | 激活 |
| Escape | 关闭 |
| Tab + Z | 显示操作 |

### 验证内容

1. 所有交互元素都可聚焦
2. 聚焦顺序合理
3. 聚焦指示器可见
4. 操作可从键盘触发
5. 键盘快捷键有文档且可发现

### 模拟器测试

在模拟器中启用 Full Keyboard Access。使用 Mac 键盘导航。

## Switch Control 测试

### 启用

**设置 > 无障碍 > Switch Control**

### 验证内容

1. 所有元素都可通过扫描到达
2. 分组元素减少扫描步骤
3. 自定义操作出现在菜单中

## Zoom 测试

### 启用

**设置 > 无障碍 > Zoom**

### 手势

| 手势 | 操作 |
|---------|--------|
| 三指双击 | 切换缩放 |
| 三指双击 + 拖动 | 调整缩放级别 |
| 三指拖动 | 缩放时平移 |

### 验证内容

1. 缩放时内容保持可用
2. 操作不会在视口外触发
3. 临时反馈（toast）可见

## Magnifier 测试

Magnifier 使用相机放大真实世界内容。如果你的应用帮助用户捕获或阅读文本，考虑：

1. 你的 UI 是否能与 Magnifier 并存
2. 避免在经常被相机预览覆盖的区域放置关键 UI 元素

## 引导式访问测试

引导式访问将设备锁定到单个应用并可限制触摸区域。如果你的应用用于教育、信息亭或专注任务：

1. 确保核心流程在没有系统手势的情况下工作
2. 避免依赖多任务手势进行关键操作
3. 如果你的 UI 依赖特定屏幕区域，验证它们在引导式访问设置中未被限制
4. 如果应用用于限时活动，考虑限时场景

## Dynamic Type 测试

### 模拟器快捷键

`Option + Command + +` 或 `-` 增大/减小文本尺寸。

### 环境覆盖

在 Xcode 的调试区域工具栏中，点击环境覆盖按钮以在不离开调试器的情况下更改 Dynamic Type。

使用此面板模拟：
- 深色模式
- 增强对比度
- 减弱动效
- 减弱透明度
- 粗体文本
- 按钮形状
- 灰度
- 不使用颜色区分

### Accessibility Inspector

**Xcode > 打开开发者工具 > Accessibility Inspector**

使用设置标签页在运行中的模拟器或设备上更改文本大小。

### SwiftUI 变体预览

在 Xcode 的画布中一次性预览所有尺寸。

### 验证内容

1. 文本随系统尺寸缩放
2. 大尺寸时不截断（除非有意的）
3. 布局为无障碍尺寸适应
4. 多行文本可读

### 双倍长度伪语言

编辑方案 > 选项 > 应用语言 > Double-Length Pseudolanguage

用更长的字符串对布局进行压力测试。

## 减弱动效测试

### 启用

**设置 > 无障碍 > 动效 > 减弱动效**

### 验证内容

1. 视差效果被禁用
2. 滑动动画变为淡入淡出
3. 自动播放动画停止或减慢

## 增强对比度测试

### 启用

**设置 > 无障碍 > 显示与文本大小 > 增强对比度**

### 验证内容

1. 文本对比度改善
2. 使用资源变体（如提供）
3. 没有元素变得不可读

## 设备测试 vs 模拟器测试

| 功能 | 设备 | 模拟器 |
|---------|--------|-----------|
| VoiceOver | 完整支持 | 有限（Mac VoiceOver） |
| Voice Control | 完整支持 | 不支持 |
| Full Keyboard Access | 完整支持 | 完整支持 |
| Switch Control | 完整支持 | 有限 |
| Dynamic Type | 完整支持 | 完整支持 |
| 触觉 | 完整支持 | 不可用 |

**建议**：在设备上测试 VoiceOver 和 Voice Control。模拟器适合布局和 Dynamic Type。

## Accessibility Inspector

### 启动

**Xcode > 打开开发者工具 > Accessibility Inspector**

### 功能

| 标签页 | 用途 |
|-----|---------|
| 检查 | 查看任何元素的无障碍属性 |
| 审计 | 运行常见问题的自动化检查 |
| 设置 | 更改 Dynamic Type、减弱动效等 |

### 连接设备

Accessibility Inspector 可与模拟器和物理设备配合使用。适用于检查其他应用。

### 通知日志

**窗口 > 显示通知**以查看无障碍通知（公告、焦点变化）。
