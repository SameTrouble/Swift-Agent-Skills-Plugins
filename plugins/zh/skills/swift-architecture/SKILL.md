# Swift 架构技能

## 概述

使用此技能为 SwiftUI/UIKit 代码库挑选最合适的 Swift 架构手册，并将其应用到用户的任务中。

如需在各手册之间快速导航，请使用 `references/_index.md`。

## 快速路径

在选择架构之前，务必先明确：
- 任务类型（新功能、重构、PR 评审、调试）
- UI 技术栈（SwiftUI、UIKit 或混合）
- 范围（单屏、多屏、整个应用）
- 状态与副作用的复杂度
- 团队熟悉度与依赖容忍度
- 需要保留的现有约定

然后：
- 如果用户明确指定了架构名称，将其作为初始候选，先做适配性检查
- 如果未指定架构，加载 `references/selection-guide.md`，根据所述约束推断最合适的方案
- 当针对单一功能且约束清晰时，选择**快速推荐模式**
- 当涉及迁移、混合架构或模块边界变更时，选择**深度重构模式**

## 快速推荐模式

适用于以下情况：
- 范围为一个功能或一个屏幕
- 约束足够清晰，可以直接推荐一个主要模式
- 用户主要需要推荐、脚手架或评审清单

交付内容：
- 适配结果（`fit` 或 `mismatch`）
- 基于请求的 1-2 条理由
- 选定的参考文件
- 针对该功能的具体结构、状态、依赖、异步与测试指导

## 深度重构模式

适用于以下情况：
- 请求涉及多个模块或多个屏幕
- 代码库已混合使用多种架构
- 用户正在从一种模式迁移到另一种模式

交付内容：
- 当前状态评估
- 目标架构推荐及适配结果（fit 或 mismatch）
- 增量迁移路径，并标明边界变更点
- 迁移过程中的风险、权衡与验证点

## 架构路由

如果用户明确指定了架构名称，将其作为初始候选，在最终确定前先做适配性检查：
- 针对 UI 技术栈适配性（SwiftUI/UIKit/混合）、状态复杂度、副作用编排需求、团队熟悉度以及现有代码库约定进行校验
- 如果适配，则按请求的架构推进
- 如果与关键约束不匹配，明确说明不匹配之处，并从 `references/selection-guide.md` 推荐最接近的替代方案
- 如果用户仍坚持使用不匹配的架构，则按风险可控的计划推进，并提前声明风险

## 分析现有代码库（视情况而定）

当代码已存在时：
- 检测当前架构与依赖注入风格
- 记录并发模型（async/await、Combine、GCD、混合）
- 使推荐与本地约定保持一致

## 护栏

- 当前的本地模式仍合理适用时，不要为了一个小功能就强制切换架构。
- 除非不匹配程度严重到足以证明变更的必要性，否则保留现有约定。
- 除非用户明确接受此权衡或代码库已在使用，否则不要引入诸如 TCA 之类的新框架依赖。
- 优先选择能干净解决请求的最小架构变更。
- 保持指导针对特定架构；除非模式之间的边界是明确的，否则不要混合手册。

## 产出具体交付物

阅读选定的架构参考，并将其指导转化为针对用户请求的交付物：

- **文件与模块结构**：针对该功能的目录布局与文件名
- **状态与依赖边界**：具体的类型、协议和注入点
- **异步策略**：取消、actor 隔离与错误路径
- **测试策略**：测什么、如何 stub 依赖、示例测试结构
- **迁移路径**（针对重构）：从当前架构迁移到目标架构的增量步骤
- **UI 技术栈适配**：针对所选架构，SwiftUI 与 UIKit 指导应在哪里区分

## 输出要求

- 使推荐范围限定在所请求的功能或评审任务内。
- 优先使用基于协议的依赖注入和显式状态建模。
- 标记现有代码中的反模式并提供直接修复方案。
- 在所有异步流程中包含取消与错误处理。
- 对于显式的架构请求，给出简短的适配结果（`fit` 或 `mismatch`）并附 1-2 条理由。
- 对于不匹配的情况，提供一个最接近的替代方案，并说明为何它更符合所述约束。
- 编写代码时，仅包含与任务相关的模式——不要倾倒整本手册。
- 默认将参考代码片段视为示例；只有当用户要求可运行代码时才补全完整的编译脚手架。
- 仅提出最少的关键阻塞问题；否则按显式声明的假设继续推进。
- 评审 PR 时，使用架构专属清单，并以行级修复指出具体违规。

## 验证清单

在最终确定之前：

1. 确认所选模式与用户的约束和技术栈匹配
2. 确认依赖注入、状态归属、副作用与测试策略都已覆盖
3. 在推荐架构变更时，显式指出迁移风险
4. 以所选架构的 PR 评审清单结尾，并适配到用户的功能
