# iOS 代码审计技能

## 运作规则

- **只读调查，单一交付物。** 不做任何代码改动——产出物是位于仓库根目录的 `CODE_AUDIT.md`。
- **每条发现项都引用 `path/to/file.swift:LINE`（或一个行范围）。** 对于 Critical 或 High 级别项目，"遍布代码库"这种说法永远不可接受。
- **严重性等级保守赋值。** Critical 意味着崩溃 / 数据丢失 / 内存损坏 / 安全暴露。不要夸大。参见下方的严重性指南。
- **每一条 Critical 主张在写入报告前都要核实。** 代理有时会夸大严重性。打开引用的文件确认缺陷真实存在。如果你通过阅读这些行无法重现该主张，就降级或丢弃。
- **按根因分组，而非按出现次数分组。** 如果一个缺失的 `@MainActor` 注解触发了七条警告，那就是一条发现项，列出七个位置，而不是七条发现项。
- **`Dead/`（或任何显式归档的目录）被排除。** 在启动代理之前，检查 `CLAUDE.md` / 项目 README 中是否有任何"请勿编辑"的目录。

## 工作流程

### 第 1 步 —— 确定代码库范围

快速测量以向代理通报情况：

```bash
find . -name "*.swift" -not -path "./.git/*" -not -path "./Dead/*" -not -path "*/Pods/*" -not -path "*/.build/*" | xargs wc -l 2>/dev/null | tail -1
find . -name "*.swift" -not -path "./.git/*" -not -path "./Dead/*" 2>/dev/null | wc -l
find . -name "*.metal" 2>/dev/null | wc -l
```

同时识别**热点文件**（最大的 LOC、核心状态）：

```bash
find . -name "*.swift" -not -path "./.git/*" -not -path "./Dead/*" -exec wc -l {} \; | sort -rn | head -10
```

如果存在 `CLAUDE.md`，请阅读它——它通常会标注核心状态文件（例如 `AppState.swift`、`CaptureState.swift`）、渲染管线，以及任何有意排除的目录。

### 第 2 步 —— 捕获编译器的事实依据

运行一次构建并提取每一条警告。这将成为审计中并发 / 弃用部分的权威输入——你应该永远不必*猜测*是否存在并发警告。

**首选方式（如果已连接 Xcode MCP）：**

```
mcp__xcode__XcodeListNavigatorIssues(tabIdentifier, severity: "warning")
```

这会返回一个结构化列表，包含 `path`、`line`、`message`、`severity`。去重（多 target 编译会产生重复项）并按根因分桶。

**备选方式（从命令行使用 `xcodebuild`）：**

```bash
xcodebuild -project <Project>.xcodeproj -scheme "<Dev scheme>" -configuration Debug build 2>&1 \
  | grep -E "warning:" | sort -u
```

如果构建是增量式的（在 2 秒内返回），它很可能跳过了大部分文件。在每个 target 下触碰一个 Swift 文件，或运行 `clean build` 以获得完整的警告集合。

### 第 3 步 —— 启动三个并行的 Explore 代理

**一条消息，三次工具调用。** 每个代理从 `references/agent-prompts.md` 获得一份聚焦的简报：

- **代理 A —— 并发与 API 现代化。** 将第 2 步捕获的警告喂给它。
- **代理 B —— 死代码、重复代码、重构候选。** 将热点列表以及 `CLAUDE.md` 中已知的过时文件喂给它。
- **代理 C —— 缺陷、逻辑错误、安全、性能。** 将热点列表以及用户特别指出的子系统（相机管线、IAP、API 客户端等）喂给它。

每份简报都强制使用逐条发现项的模板（见下文）。不要让代理返回摘要——坚持要求具体的 file:line 发现项。

### 第 4 步 —— SwiftUI 专家审查（如果项目使用了 SwiftUI）

如果项目包含 SwiftUI 视图，在代理返回*之后*单独调用 `swiftui-expert-skill`，作用范围限定于 `Views/`（或 SwiftUI 界面所在之处）以及共享的 UI 辅助工具。具体询问：

- `@State` / `@Bindable` / `@Observable` 的误用
- 修饰符顺序错误
- 应当上提的视图 body 内的工作（格式化器、解码器、排序）
- 无障碍 / Dynamic Type / 深色模式的缺失
- 热门叶子视图缺少 `Equatable`
- Liquid Glass / iOS 26 的采用机会（仅在用户要求时）

你可能需要自己阅读几个视图文件，以确认该技能识别出的模式。

### 第 5 步 —— 核实 Critical 发现项

在撰写报告之前，**打开每一条 Critical 级别发现项所引用的行**并确认：

- 代码与代理描述的一致。
- 影响主张是真实的（例如，如果代理说"内存损坏"，缓冲区是否真的尺寸不足？追踪一下数学运算。）。
- 推荐的修复方案合理。

这一步在以往的运行中曾捕获过幻觉的严重性。对那些经不起推敲的项目进行降级或丢弃。**永远不要传播一条你本人未曾核实的 Critical 发现项。**

### 第 6 步 —— 综合 `CODE_AUDIT.md`

使用 `references/report-template.md` 中的骨架。

**强制要求：渲染出的报告必须在每个标题中包含章节编号。** 这不是装饰性的——它是报告的主要接口。用户会提交诸如"修复 §5.4"和"§3.1 做完了吗？"这样的 issue，如果标题没有编号，他们就无法做到这一点。**生成无编号的报告是一种缺陷，而不是风格选择。** 没有编号的报告将被拒绝。

编号规则：

- **顶级章节编号为 `## 1.` 到 `## 12.`** —— 完全如下方所列，按此顺序，即使某个章节没有发现项（在这种情况下，在标题下写"_无发现项。_"，而不是省略该章节）。
- **每条发现项都是一个带编号的子章节**，形式为 `### N.M <简短标题>`，其中 `N` 是父章节编号，`M` 从 1 开始递增。示例：`### 5.1 Bundle.main.url 上的强制解包`，`### 5.2 缺少 .limited Photos 授权处理`。**永远不要**在没有 `N.M` 前缀的情况下将发现项写成 `### <简短标题>`。
- **编号在编辑过程中保持稳定。** 如果某条发现项在修订期间被移除，保留编号并在正文处写 `_REMOVED: <原因>_` —— 不要对存活的发现项重新编号，否则所有对"§5.4"的外部引用都会失效。
- **执行摘要条目**（§1）是一个有序列表 `1.`、`2.`、……，引用底层编号的发现项（例如，"**[Critical] Bundle.main.url 上的强制解包** —— §5.1 —— `path:line`"）。
- **核实条目**（§12）通过其子章节编号引用发现项，例如，`- **§5.1** —— 打开 \`path\`，第 42-47 行。`。不要留下模板中的 `<N.M>` 占位符——要把它们填上。

顶级章节，按顺序排列：

1. **执行摘要** —— 5-10 条最高影响的发现项，每条一行，带严重性标签和 §N.M 回溯引用。
2. **快速收益** —— ≤30 分钟的修复（删除过时文件、移除调试 `print`、修复 unused-let 警告、添加无障碍标签）。
3. **并发**
4. **API 现代化** —— 弃用项、iOS-17+ 的替代 API
5. **缺陷 / 逻辑错误**
6. **安全**
7. **性能**
8. **SwiftUI / UI**
9. **死代码 / 重复代码 / 重构**
10. **横切建议** —— 值得在整个仓库范围应用的模式
11. **未审计的内容** —— 显式的范围外清单
12. **核实** —— 针对每条 Critical/High，证明该主张的确切代码行

逐条发现项模板（注意开头的 `N.M`）：

```markdown
### N.M <简短标题>
- **位置：** `path/to/file.swift:LINE-LINE`
- **问题：** <用一句话描述观察到的问题>
- **原因：** <影响 / 为什么重要>
- **行动：** <推荐的修复方案；不要写代码，引用模式>
- **严重性：** Critical | High | Medium | Low
```

总体篇幅要全面但可扫读——目标 50-100 条发现项。如果某个类别有许多实例，在同一个标题下把相似的出现项分组（列出最具体的 5-10 个示例加上其余的计数）。

## 严重性指南

- **Critical** —— 可能导致崩溃、数据丢失、内存损坏、安全暴露，或把错误的服务器 URL 发到生产环境。在赋这个等级之前，自己打开引用的行确认一下。
- **High** —— 用户能触发的真实缺陷；在未来某个 Swift 模式下会变成错误的编译器警告；正在被积极移除的已弃用 API；架构性的并发问题。
- **Medium** —— 性能 / 质量问题；重构候选；没有功能性缺陷的缺失现代化。
- **Low** —— 命名、外观、代码风格、单次出现的清理。

## 质量门槛（交付前的最终检查）

- [ ] **每个顶级章节都编号为 `## 1.` 到 `## 12.`**，且每条发现项都是带编号的子章节 `### N.M <标题>`。扫一遍渲染出的报告——如果任何标题缺少编号，在交付前修复它。没有编号的报告将被视为缺陷而拒绝。
- [ ] **执行摘要条目通过 §N.M 引用发现项**，核实条目也是如此——模板中没有留下字面的 `<N.M>` 占位符。
- [ ] 每条 Critical 和 High 发现项都有确切的行范围。没有"遍布代码库"。
- [ ] 并发发现项交叉引用了实际的第 2 步警告列表。
- [ ] 至少 3-5 条 Critical 发现项已通过打开引用文件进行了抽查核实。
- [ ] 发现项按根因分组（一个注解 → 多条警告 = 一条发现项）。
- [ ] "未审计的内容"章节是显式的，以便用户了解缺口。
- [ ] 报告包含 ≤ ~80 条不同的发现项——更大的报告会失去可执行性。
- [ ] 报告本身不包含任何代码——建议描述的是模式，而非实现。
- [ ] 用户可以把任何一条发现项单独拉到一个独立任务中，而无需重新解释上下文。

## 技能输出

始终写入位于**仓库根目录**（用户项目的工作目录）的 `CODE_AUDIT.md`。如果已存在则覆盖——本技能产出的是快照，不是追加日志。如果用户想要历史记录，可以在调用之前先 `git mv` 之前的版本。

## 本技能不涵盖的内容

- **Metal 内核 / 领域特定代码的算法正确性。** 只呈现显而易见的问题；深度的算法审查不在范围内。
- **构建设置、scheme 配置、Xcode 项目结构。** 超出共享 scheme 中可见内容的范围。
- **第三方依赖的内部实现。** SPM / CocoaPods 包被视为黑盒。
- **测试覆盖评估。** 快速扫一眼测试 target 可以；深度的测试审查是单独的技能。
- **本地化正确性。** 审计可以标注未翻译的字符串，但不评估措辞。
- **性能剖析。** 本技能识别*潜在的*热路径（每帧分配、过宽的状态等），但不运行 Instruments trace。为此，请单独调用 `swiftui-expert-skill` 及其 trace 工具。
